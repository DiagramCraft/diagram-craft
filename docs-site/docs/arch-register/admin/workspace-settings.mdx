Workspace Settings is the main governance screen for a workspace.

## Covered Areas

The implemented settings area includes:

- general workspace identity
- lifecycle
- model overview
- entity types (schemas)
- document types and templates
- roles and permissions
- global permissions
- teams
- members
- AI
- export and import
- analytics
- audit log
- webhooks
- job monitoring
- danger zone

## General

Use general settings to manage the basic workspace identity such as name, slug, description, and visual identity.

## Lifecycle

Lifecycle settings define the allowed lifecycle states for entities in the workspace. They control classification and
filtering, not access.

One state can be designated as the workspace's deprecated state. Finalizing an
[entity deprecation](../use/governance) moves the entity to that state.

## Governance Policies

Approval and deprecation requirements are configured per entity type, not workspace-wide. On each entity type in
**Entity Types**, administrators can set:

- **Entity approval policy** — require approval before entity changes take effect
- **Entity deprecation** — require a deprecation proposal to retire entities of this type

See [Governance and Approvals](../use/governance) for how these workflows behave for users.

## Document Types And Templates

Document Types & Templates defines the structured Markdown documents available in the workspace. Document types
provide fields and validation rules; templates provide reusable Markdown bodies and metadata defaults. Project owners
can also create project-specific Markdown templates from project settings.

See [Document Types And Templates](document-types-templates) for the full configuration and usage workflow.

## Analytics, Audit, And Danger Zone

These three sections exist for different operational jobs:

- **Analytics** for workspace-wide coverage and completeness views
- **Audit log** for change inspection
- **Danger zone** for irreversible operations

Keep them conceptually separate. They solve different problems.

## Webhooks And Job Monitoring

Two operational sections support integrations and background work:

- **Webhooks** configures outbound webhooks for workspace events and shows delivery behavior, so external systems
  can react to changes in Arch Register.
- **Job monitoring** shows scheduled jobs, job servers, and run history, with cancellation available for supported
  jobs.
