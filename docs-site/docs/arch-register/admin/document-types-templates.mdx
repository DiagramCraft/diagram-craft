# Document Types And Templates

Document types add structured metadata to Markdown documents. They let a workspace standardize recurring
documentation without giving up the flexibility of Markdown for the document body.

A typed document has:

- a Markdown body
- one document type
- values for the fields defined by that type
- optional links to entities and other documents

## Where To Manage Them

Open **Workspace Settings → Document Types & Templates**. The screen has two tabs:

- **Types** defines the fields and validation rules for typed documents.
- **Templates** defines reusable Markdown bodies and metadata defaults for workspace-wide documents.

Project owners can also manage project-specific Markdown templates from **Edit project → Project Markdown
templates**. Workspace templates are available throughout the workspace; project templates are available when
creating documents inside their project.

## Define A Document Type

Create a type with a name, description, color, and icon. Add fields to describe the information that should be
captured alongside the Markdown body.

| Field type | Use it for |
| --- | --- |
| Text | Short single-line values |
| Long text | Longer free-form values |
| Boolean | True/false values |
| Date | Calendar dates |
| Number | Numeric values |
| Enum | One value from a defined set of options |
| Entity link | Links to one or more entities |
| Document link | Links to one or more documents |

Each field has a stable field ID, a display name, and a requirement:

- **Required** blocks saving until a value is supplied.
- **Expected** shows a warning when the value is missing but does not block saving.
- **Optional** does not require a value.

Link fields can also define minimum and maximum cardinality. Enum fields must define their allowed options.
Values are checked against the field type when a document is saved.

### AI Actions

A document type can define interactive AI actions and metadata generators. Actions are named, predefined prompts that
use the document body, metadata, type, and location with read-only tools. Interactive actions can be run by anyone who
can view a typed document; metadata generators run automatically after document changes. Neither action kind can
modify documents, metadata, or entities through its AI tools.

Actions are shown by name and type when editing a document type. Select **Add AI action** or **Edit** to open the action
dialog, where you can set the name, kind, prompt, output field for metadata generators, and enabled state.

Before saving or enabling an action, administrators can choose an existing accessible document of the same type and
select **Test action**. Testing uses the current unsaved action draft and returns the complete model output, provider,
model, duration, parsed metadata value when applicable, and any actionable parsing, validation, permission, or tool
errors. Testing does not change the document, metadata, revisions, or generation schedules.

AI actions require workspace [AI configuration](ai-configuration) to be in place. See
[AI Assistant and Extract](../use/ai-assistant-extract) for the reader-facing workflow.

### Evolving A Type

Document types are designed to evolve without invalidating document history:

- A field already used by documents keeps its ID and value type.
- Removing a field that is already used retires it, preserving the field for existing documents and revisions.
- A type used by documents, revisions, or templates cannot be permanently deleted; archive it instead.
- Archived types remain visible on existing documents and in history, but are not offered for new documents.

When changing the type of an existing document, use the migration flow. The document must be brought into
compliance with the new type, including removing metadata that is not defined by that type.

## Create A Document Template

A document template belongs to one document type and contains:

- a template name
- a Markdown body
- optional structured metadata defaults

Use `{{title}}` in the body where the new document name should be inserted. When someone creates a document from
the template, that token is replaced with the document name.

Metadata defaults use the field IDs from the selected document type. They must use valid values for those fields;
enum defaults must be one of the configured options, and links must contain valid entity or document IDs.

Workspace templates are created in the **Templates** tab of Workspace Settings. Project templates are created in
the project settings dialog and are restricted to that project. Archived templates remain available for existing
documents and history, but are not offered when creating new documents.

## Create A Typed Document

From workspace, project, or entity content:

1. Choose **New wiki page**.
2. Enter the document name.
3. Select a **Document type**, or choose **Untyped Markdown** for a legacy-style document.
4. Optionally select a **Template**.
5. Complete the document body and any required properties, then save.

Selecting a template also selects its document type and fills the body and metadata defaults. Without a template,
the document starts with a blank Markdown body and the selected type's fields.

While editing, the **Properties** panel shows the fields for the selected type. Entity and document link fields
provide link pickers, and required or expected fields show their validation state. Read-only documents display the
same metadata without editing controls.
