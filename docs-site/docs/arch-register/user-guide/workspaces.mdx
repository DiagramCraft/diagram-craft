# Workspaces

Workspaces are the top-level boundary for architecture data in Arch Register.

Each workspace keeps its own schemas, entities, projects, teams, members, and settings separate from other workspaces. That makes it the main unit for both isolation and administration.

## What a workspace contains

A workspace organizes the parts of Arch Register that belong together:

- **Schemas** define the entity types, fields, and enums used in that workspace.
- **Entities** are records created from those schemas.
- **Projects** collect diagrams, folders, templates, and related files.
- **Teams** own content and provide an ownership layer separate from workspace membership.
- **Members** define who belongs to the workspace and which workspace role they have.
- **AI settings** configure the assistant and extraction features for that workspace.

## How the pieces fit together

Think of the workspace as the container and the other concepts as the layers inside it:

- **Schemas** shape the model.
- **Entities** are the data created from that model.
- **Projects** organize diagrams and documentation around work in progress.
- **Teams** own records and help distribute access.
- **Workspace roles** control what a person can do across the whole workspace.
- **AI settings** configure workspace-wide assistant behavior.

That separation is intentional:

- content lives in entities and projects
- governance lives in roles, teams, and members
- workspace settings control structure, access, and operational behavior

Common workspace flows are usually:

1. Open the workspace selector in the top bar.
2. Switch to another workspace when you need a different data boundary.
3. Create a new workspace from the selector when you need a new isolated area.
4. Open `Settings` from the workspace shell when you need to adjust identity, model, access, or AI configuration.

## Roles and access

Workspace access is not all-or-nothing.

- **Workspace roles** determine what a member can do in the workspace shell and admin surfaces.
- **Team roles** determine access and responsibility within content owned by a team.
- **Global permissions** control platform-wide capabilities that are outside a single workspace.

If a user cannot see a settings section, that section is usually hidden by permissions rather than missing from the product.

## Defaults and limitations

What users should expect by default:

- the current workspace is the one shown in the shell and top bar
- workspace creation opens the new workspace and updates the selector list
- not every workspace member can access every settings area
- AI features only appear when AI is enabled for the workspace

Other practical limits:

- workspace roles do not replace team roles
- lifecycle states classify records, but they do not grant access
- settings pages are separated from content editing so governance changes stay in one place
