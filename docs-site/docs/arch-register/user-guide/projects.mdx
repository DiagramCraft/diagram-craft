# Projects and diagrams

Projects are workspace-scoped containers for diagrams, wiki pages, attachments, and folders.
They sit inside the workspace boundary, but the workspace still owns the people, teams, schemas, and permissions that
decide who can see or change project content.

## Workspace vs project

- The **workspace** is the top-level boundary for configuration, members, roles, schemas, and shared access rules.
- A **project** is a content boundary for related files, diagrams, and notes.
- Projects can reference entities and schemas when a diagram or document needs architectural context.
- Project access is owned by the project’s owner team, so project actions are controlled through the workspace
  permission model rather than through the project itself.

## What a project contains

- Folders for organizing related content.
- Diagrams for architecture views and workflows.
- Wiki pages for project notes, decisions, and supporting documentation.
- Attachments and uploaded files.
- Templates that can be reused when creating new diagrams.
- Real-time collaboration for shared editing.

## Project browser

Use the project browser to review project metadata, switch between active and archived projects, navigate into folders,
and open content.

Pinned projects stay near the top of the Projects tab. Draft and active projects are listed separately from archived
projects, which appear under the Archive tab.

## Folders and nested diagrams

Projects use folder paths to keep large sets of diagrams and notes manageable.
Folders can be nested, and the sidebar shows the tree so you can move between sibling and child content quickly.

Use folders when you want to group content by system, delivery stream, or ownership area:

- keep a root level for a small project with only a few files
- create subfolders when one project needs separate streams or phases
- move diagrams and wiki pages into folders when the project starts to sprawl

From the item menus you can create folders, rename them, move files between folders, or move content back to the root.

## Creating diagrams

1. Open a project.
2. Select `New` and then `New diagram`.
3. Leave `Blank canvas` selected to start from an empty diagram.
4. Enter a diagram name.
5. Select `Create diagram`.
6. Open the new diagram card to launch the editor.

Use the same dialog to create diagrams from templates when a project template or workspace template is available.

## Project lifecycle

Projects currently use four statuses:

- `Draft`
- `Active`
- `Complete`
- `Cancelled`

Use `Edit` to change project metadata, including status, owner, color, and target date.
Archived projects are the ones marked `Complete` or `Cancelled`; they move into the Archive tab in the project
sidebar.

To restore an archived project, edit the status back to `Draft` or `Active`.
If you need to permanently remove a project, the Edit dialog includes a delete action that removes the project and all
of its files.

Pinned projects stay easy to find in the Projects tab, and you can pin or unpin them from the project header when you
have edit access.

## Permissions

Project actions are controlled by the project owner team and the workspace permission model.
The UI exposes three main project capabilities:

- `canEdit` for editing project metadata, pinning, and changing status
- `canManageFiles` for creating folders, uploading files, adding diagrams, adding wiki pages, and managing file menus
- `canDelete` for permanently deleting the project

If you need the broader permission setup, see the [Administration Guide](../admin-guide).
That page covers workspace roles, team roles, and the role/permission mapping that project access depends on.

## Templates

Templates help standardize recurring diagram types.
Project templates are available inside the current project.
Workspace templates are available wherever the template rules allow them.

You can promote a diagram to a template from its card menu, then create future diagrams from that starting point.

## Search and filtering

The filter box inside a project only narrows the content you are already looking at.
It matches file names in the current project or folder, which makes it useful for finding a diagram or wiki page quickly.

For broader discovery, use workspace search.
Workspace search looks across projects, files, entities, and schemas, and it includes the project name and scope when a
result comes from project content.

Use the project filter when you know which project you are in.
Use workspace search when you do not know where the content lives yet.

## Collaboration

When multiple users open the same diagram, Arch Register connects them to the same collaborative session.
The presence toolbar shows who is currently in the file, and edits sync in real time.

## Common workflows

- Create a blank diagram for a new initiative, then refine it in the editor.
- Turn a finished diagram into a project template so future work starts from the same baseline.
- Keep related diagrams in folders to make large projects easier to scan.
- Archive a completed project by changing its status to `Complete` or `Cancelled`, then find it later in the Archive
  tab.
- Open the same diagram with a teammate to review changes together.

If you want to document the project itself, use [wiki pages](wiki) inside the project instead of keeping notes in a
separate tool.
